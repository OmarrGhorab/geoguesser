# Tailwind CSS v4

## Concepts

Tailwind CSS v4 uses CSS-first configuration and design tokens exposed as CSS variables through `@theme`. Use Tailwind utilities for layout and styling, CSS variables for theme values, and logical properties for RTL support.

Why this exists: utility classes keep styling local and consistent, while v4 theme variables make tokens usable by both utilities and runtime CSS.

## Best Practices

- Use Tailwind CSS v4.
- Define design tokens in CSS with `@theme`.
- Use CSS variables for app themes and shadcn colors.
- Prefer utility classes over ad hoc CSS files.
- Use logical spacing and alignment where RTL matters.
- Keep color, radius, spacing, and typography tokenized.
- Use `cn()` helpers for conditional classes.

## Anti-Patterns

- Tailwind v3 `tailwind.config.js` assumptions in v4 projects.
- One-off arbitrary values for every component.
- Hardcoded hex colors in JSX.
- CSS Modules unless an existing project explicitly requires them.
- Overriding shadcn components with brittle selectors.
- Palette choices that depend on color alone for state.

## Common Mistakes

- Defining CSS variables that do not map to Tailwind theme variables.
- Forgetting dark mode variables.
- Using physical left/right classes in RTL-sensitive UI.
- Removing focus ring utilities.
- Letting class strings become unreadable instead of extracting variants.
- Styling based on DOM structure generated by Radix instead of documented APIs.

## Production Examples

```css
/* app/globals.css */
@import "tailwindcss";

@theme {
  --color-brand-50: oklch(0.98 0.02 210);
  --color-brand-600: oklch(0.55 0.18 230);
  --radius-control: 0.5rem;
}

:root {
  --background: oklch(1 0 0);
  --foreground: oklch(0.18 0 0);
}

[data-theme="dark"] {
  --background: oklch(0.16 0 0);
  --foreground: oklch(0.96 0 0);
}
```

```tsx
export function Stat({ label, value }: { label: string; value: string }) {
  return (
    <section className="rounded-md border bg-background p-4 text-foreground">
      <p className="text-sm text-muted-foreground">{label}</p>
      <p className="mt-2 text-2xl font-semibold">{value}</p>
    </section>
  )
}
```

## Folder Organization

```text
app/globals.css
components/ui/
lib/utils.ts
features/*/components/
```

Keep global CSS limited to imports, tokens, base layer rules, and app-wide variables.

## TypeScript Examples

```ts
// lib/utils.ts
import { clsx, type ClassValue } from 'clsx'
import { twMerge } from 'tailwind-merge'

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs))
}
```

```tsx
import { cn } from '@/lib/utils'

export function Badge({
  tone,
  children,
}: {
  tone: 'neutral' | 'success' | 'danger'
  children: React.ReactNode
}) {
  return (
    <span
      className={cn(
        'inline-flex rounded-sm px-2 py-1 text-xs font-medium',
        tone === 'success' && 'bg-emerald-50 text-emerald-700',
        tone === 'danger' && 'bg-red-50 text-red-700',
        tone === 'neutral' && 'bg-muted text-muted-foreground'
      )}
    >
      {children}
    </span>
  )
}
```

## Performance Considerations

- Utilities avoid shipping per-component CSS bundles.
- Keep class generation predictable so unused CSS stays small.
- Avoid runtime style objects for static styling.
- Prefer CSS transforms and opacity for animated states.
- Avoid layout shifts by defining stable sizes with `size-*`, aspect ratios, and min/max constraints.

## Security Considerations

- Do not build class names from unsanitized user input.
- Avoid injecting untrusted CSS variables.
- Keep theme customization controlled by known tokens.
- Do not use CSS to hide unauthorized data; do not render it.
- Review arbitrary values in security-sensitive UI.

## Accessibility Considerations

- Preserve focus-visible styles.
- Meet color contrast requirements in every theme.
- Use motion-safe and motion-reduce variants.
- Use logical properties for RTL.
- Do not communicate state by color alone.
