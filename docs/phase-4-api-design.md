# Phase 4 API Design

## Purpose

This document defines the REST API contract before implementation. It aligns the Phase 2 architecture and Phase 3 database model with a versioned OpenAPI 3.1 specification.

The machine-readable OpenAPI contract lives at:

```text
backend/openapi/openapi.yaml
```

## API Principles

- Use RESTful resources under `/api/v1`.
- Use JSON request and response bodies.
- Use HTTP-only cookies for authentication.
- Use short-lived JWT access tokens and rotated refresh tokens.
- Use CSRF protection for unsafe cookie-authenticated requests.
- Keep exact location coordinates hidden until a guess is submitted or the round expires.
- Keep handlers thin. Handlers decode, validate, call services, and encode responses.
- Keep business rules in services.
- Use idempotency keys for retryable writes such as guess submission, room creation, checkout creation, and payment webhooks.
- Never return raw database models. Return stable DTOs.
- Return consistent error envelopes for every failure.

## Base URLs

| Environment | Base URL |
| --- | --- |
| Local through Nginx | `http://localhost/api/v1` |
| Local API direct | `http://localhost:8080/api/v1` |
| Production | `https://<domain>/api/v1` |

## Authentication Model

The API supports both guest gameplay and registered accounts.

Registered users authenticate with:

- `access_token`: HTTP-only secure cookie, short-lived.
- `refresh_token`: HTTP-only secure cookie, path-scoped to refresh endpoint.
- `csrf_token`: readable cookie or header-bound token for unsafe methods.

Guest players use a signed guest session cookie generated by the backend. Guest sessions can play games and join rooms, but cannot access persistent profile, billing, friends, or registered-user leaderboards.

## Authentication Classes

| Class | Meaning |
| --- | --- |
| Public | No session required. Used for public maps, health, and public leaderboards. |
| Optional session | Guest or registered session accepted. Used for starting games and joining rooms. |
| Authenticated | Registered user required. Used for profile, billing, saved stats, friends, and account features. |
| Host | Current room host required. Used for room settings, start, and kick actions. |
| Admin | Admin or moderator role required. Used later for location/map administration. |
| Webhook | External provider signature required. Used for payment webhooks. |

## Error Format

Every error response uses the same shape.

```json
{
  "error": {
    "code": "validation_failed",
    "message": "The request body is invalid.",
    "request_id": "req_01J...",
    "fields": [
      {
        "name": "email",
        "code": "invalid_email",
        "message": "Email must be valid."
      }
    ]
  }
}
```

Rules:

- `code` is stable and machine-readable.
- `message` is safe for users but not used for program logic.
- `request_id` is included whenever available.
- `fields` appears only for validation errors.
- Internal errors never expose SQL, provider secrets, stack traces, tokens, or raw infrastructure messages.

## Common Status Codes

| Status | Use |
| --- | --- |
| `200` | Successful read or command with response body. |
| `201` | Resource created. |
| `204` | Successful command with no body. |
| `400` | Invalid request shape or validation error. |
| `401` | Missing or invalid authentication. |
| `403` | Authenticated but not allowed. |
| `404` | Resource not found or intentionally hidden. |
| `409` | Conflict, duplicate submission, stale room state, or already-started game. |
| `422` | Valid JSON but invalid domain transition. |
| `429` | Rate limited. |
| `500` | Unexpected server failure. |

## Pagination Format

List endpoints use cursor pagination by default.

Request:

```text
GET /api/v1/leaderboards/global?limit=50&cursor=...
```

Response:

```json
{
  "data": [],
  "page": {
    "limit": 50,
    "next_cursor": "eyJzY29yZSI6..."
  }
}
```

Rules:

- `limit` defaults to 20.
- `limit` maximum is 100.
- Sort fields must be allowlisted.
- Cursors must be opaque to clients.
- Total counts are returned only where cheap and useful.

| `POST` | `/auth/login` | Public | Login and set auth cookies. |
| `POST` | `/auth/logout` | Authenticated | Revoke current refresh session and clear cookies. |
| `POST` | `/auth/refresh` | Refresh cookie | Rotate refresh token and issue new access token. |

| Method | Path | Auth | Purpose |
| --- | --- | --- | --- |

## Security Rules

- All unsafe methods using cookies require CSRF validation except verified webhooks.
- Login, register, refresh, room creation, room join, queue join, and guess submission are rate limited.
- Payment webhooks require provider signature verification and idempotency.
- Room codes must not be enumerable.
- `round.location_id`, `location.latitude`, and `location.longitude` are never exposed before reveal.
- Guest session identifiers must be signed, random, and stored only as hashes when persisted.
- Admin-only endpoints must check role in service code, not only middleware.

## Implementation Notes

- Use OpenAPI schemas as DTO contract names, not GORM model names.
- Generate or validate Go DTOs against the OpenAPI spec during CI once tooling is chosen.
- Serve Swagger UI only in local development or protected internal environments.
- Keep `/metrics` restricted to Prometheus/internal network.
- Prefer `POST /games/{gameId}/rounds/{roundId}/guesses` over a short `/games/{id}/guess` path because the round is an explicit resource and the one-guess-per-round rule is clearer.

## Open Questions

- Should guest users be allowed to create private rooms?
- Should public leaderboards include guests or only registered users?
- Which payment provider will define checkout and webhook details?
- Will the MVP use WebSocket events immediately, or begin with HTTP polling?
- What exact imagery provider fields must be exposed in `RoundMedia`?

## Phase 4 Exit Criteria

Phase 4 is ready for implementation when:

- Endpoint list is approved.
- OpenAPI paths and schemas are approved.
- Error format is accepted.
- Authentication and CSRF rules are accepted.
- Guest versus registered permissions are clear.
- Hidden coordinate behavior is accepted.
- MVP endpoints are identified for first backend implementation.
